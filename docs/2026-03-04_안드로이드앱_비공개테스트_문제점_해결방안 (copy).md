## 앱 전체 구조 (비개발자용 설명)

이 앱은 두 층으로 이루어져 있습니다.

```
[바깥층: 안드로이드 앱 껍데기]
  - Google Play에 올라간 설치 파일 (.aab)
  - 카메라·마이크·위치 같은 하드웨어 권한을 안드로이드에 요청
  - 내부에 브라우저 창(WebView)을 하나 띄움

[안층: 웹 앱 (실제 기능이 모두 여기 있음)]
  - 서버에서 실시간으로 내려받아 실행
  - 화면 전환·AI 가이드 생성·TTS 음성 재생 모두 여기서 처리
  - 가이드북 공유 페이지도 여기서 생성
```

**화면 종류 (앱 안에서 전환되는 화면들):**

| 화면 이름 | 사용자가 보는 것 |
|----------|----------------|
| 언어선택 (landingPage) | 앱 최초 실행 시 언어 선택 |
| 기능소개 (featuresPage) | 온보딩 + "바로 시작하기" 버튼 |
| 메인 화면 (mainPage) | 카메라·사진 업로드·음성 입력 (핵심 기능) |
| 결과 화면 (detailPage) | AI가 생성한 가이드 텍스트 + 음성 재생 화면 |
| 보관함 (archivePage) | 상단 추천갤러리 고정 + 중간 보관함 스크롤 |
| 설정 (settingsPage) | 언어 변경, 알림 등 설정 |

## 핵심 구조 원칙: 독립 페이지 vs 앱 내부 모달

### 비유: 네이티브 앱의 팝업과 같은 개념

네이티브 앱에서 이벤트/공지 팝업이 단순 닫기로 작동하는 이유:
- 팝업은 메인 화면 **위에 올라온 레이어**
- 닫으면 메인 화면이 그대로 살아있음
- 메인 화면의 세션·권한·상태가 전혀 건드려지지 않음

이 앱의 독립 페이지도 동일한 개념입니다. **메인 앱(WebView)이 백그라운드에 살아있는 상태에서 독립 창이 열리고 닫힌다.**

### 독립 페이지 7개 (별도 창 — 닫기 = window.close())

| 페이지 | 닫기 버튼 | 수정 전 | 수정 후 |
|--------|----------|--------|--------|
| `/profile.html` | 상단 X 버튼 | `window.close()` | ✓ 정상 |
| `/admin-dashboard.html` | 상단 닫기 버튼 | `window.close()` | ✓ 정상 |
| `/user-guide.html` | 뒤로가기 버튼 | `window.close()` | ✓ 정상 |
| `/s/{id}` 일반 공유 페이지 | 우측 상단 X | `window.close()` | ✓ 정상 |
| `/s/{id}` **추천갤러리 공유 페이지** | 좌측 상단 ← | `window.location.href='/#archive'` | **→ `window.close()`** |
| **Google 인증 완료 페이지** | 인증 완료 버튼 | opener 없으면 `'/#archive'` 리다이렉트 | **→ `window.close()`** |
| **Kakao 인증 완료 페이지** | 인증 완료 버튼 | opener 없으면 `'/#archive'` 리다이렉트 | **→ `window.close()`** |

### 앱 내부 모달 7개 (index.html 안 오버레이 — 이미 정상)

| 모달 | 역할 |
|------|------|
| `authModal` | 로그인 모달 |
| `shareModal` | 공유 처리 모달 |
| `notificationModal` | 알림 모달 |
| `infographicModal` | 인포그래픽 모달 |
| `videoModal` | 동영상 모달 |
| `user-admin-auth-modal` | 관리자 인증 모달 |
| `user-qr-code-modal` | QR 코드 모달 |

## 문제점 A: 앱 재시작 시 로그인이 풀림 (인증 정보 디바이스 저장 미흡)

### 현상
앱을 껐다 켜거나 WebView가 갱신되면 서버 세션 쿠키가 유실되어 로그인 화면이 다시 나타남.

### 원인
현재 로그인 상태는 서버 세션(쿠키)에만 의존합니다. 안드로이드 WebView는 앱 재시작 시 쿠키가 초기화되는 경우가 있어 로그인이 풀린 것처럼 보입니다.

### 해결 방향
로그인 성공 시 사용자 식별 정보(이름, 이메일, 제공자)를 디바이스 localStorage에도 저장합니다.

- **앱 시작 시**: localStorage 캐시 먼저 확인 → 로그인 화면 없이 바로 메인 화면 표시
- **백그라운드에서 서버 검증**: 서버가 확인해주면 그대로 사용. 실패 시에만 localStorage 삭제 후 로그인 화면 표시
- **저장 정보**: 표시용 정보만 (id, 이름, 이메일, 로그인 제공자). 민감한 토큰은 저장하지 않음
- **로그아웃 시**: localStorage 캐시도 함께 삭제

### 수정 후 예상 시나리오
- 앱 최초 설치 후 로그인: 정상 로그인, localStorage에 저장
- 앱 재시작: localStorage에서 바로 인식 → 로그인 화면 없이 메인 화면 표시
- 서버 점검 중: 캐시로 화면 표시 유지 (기능 제한은 있을 수 있음)

## 문제점 1: 앱을 껐다 켜거나 언어를 바꾸면 JS 파일이 구버전으로 실행됨

### 현상
개발팀이 서버에 새 파일을 올려도 앱이 구버전 파일을 계속 실행함. 로그인 관련 코드도 구버전이 실행되어 로그인이 풀린 것처럼 보일 수 있음.

### 원인
> 비유: 마트 직원이 어제 찍어둔 오래된 사진으로 오늘 손님을 안내하는 상황

앱에 **"오래된 파일 저장고"(서비스워커 캐시)**가 있어서, 서버에 최신 파일이 있어도 저장고의 구버전을 계속 꺼내 씁니다.

### 해결 방향
저장고 규칙을 변경합니다.

- **현재 규칙:** 저장고에 있으면 저장고 것을 먼저 씀 (구버전 계속 사용)
- **바꿀 규칙:** 항상 인터넷(서버)에서 최신 파일을 받아 씀. 인터넷이 안 될 경우에만 저장고 사용

오프라인에서도 볼 수 있어야 하는 **공유 가이드북 페이지**(`/s/*`)는 기존 방식(저장고 우선) 그대로 유지.

### 수정 후 예상 시나리오
- 앱을 껐다 켜도: 항상 최신 파일 실행
- 언어 변경 후: 최신 파일 실행
- 오프라인 공유 가이드북: 변경 없이 정상 작동

## 문제점 2: 카메라·마이크·위치 권한을 앱 켤 때마다 다시 물어봄

### 현상
처음 앱 설치 후 카메라·마이크·위치 권한을 허용했는데, 앱을 끄고 다시 켜면 또 허용 팝업이 뜸.

### 원인
> 비유: 매장 입구에서 신분증을 확인하는 직원이, 한 번 확인했다는 사실을 메모하지 않아 같은 손님에게 매번 신분증을 요구하는 상황

안드로이드 앱 껍데기에 "권한을 한 번 허용받으면 기억해라"는 코드가 없습니다.

### 해결 방향
앱 껍데기 코드에 "권한을 허용받으면 기억해서 다음엔 묻지 말라"는 처리를 추가합니다.

**중요:** 이 수정은 앱 껍데기(설치 파일) 변경이므로, **Google Play에 새 버전(versionCode 4)을 다시 올려야 합니다.**
이번 작업에서는 사용자 승인 후 별도로 진행 예정.

### 수정 후 예상 시나리오
- 앱 최초 설치 후: 권한 팝업 1회 표시
- 허용 후 앱 재시작: 팝업 없이 바로 사용 가능

## 문제점 3: 추천갤러리 공유 페이지에서 닫기 버튼을 누르면 엉뚱한 화면이 나옴

### 현상
추천 가이드 공유 페이지를 보고 왼쪽 상단 ← 버튼을 눌렀을 때, 보관함이 아닌 메인 화면이나 로그인 화면으로 이동됨.

### 원인
> 비유: 도서관에서 책을 읽다가 출구 표지판을 따라갔더니 주차장이 아닌 옆 건물로 나오는 상황

추천갤러리 공유 페이지는 메인 앱 위에 올라오는 **독립 팝업 창**입니다. 그런데 닫기 버튼이 "창 닫기" 대신 **"보관함 주소로 새로 이동하라"** 는 명령을 수행합니다.

안드로이드 WebView 안에서 "새 주소로 이동"을 하면:
1. 앱 내부 화면 히스토리가 초기화됨
2. 새 파일을 다시 불러오면서 구버전 캐시 파일이 로드될 수 있음
3. 로그인 정보까지 영향받을 수 있음

### 해결 방향
닫기 버튼을 **"창 닫기"(`window.close()`)** 로 변경합니다.
창이 닫히면 백그라운드에 살아있던 메인 앱이 그대로 표시됩니다. 어떤 화면에서 열었든 그 화면으로 자연스럽게 복귀됩니다.

### 수정 후 예상 시나리오
- 보관함 → 추천갤러리 → ← 버튼: 보관함으로 정확히 복귀
- 어떤 화면에서 열든: 그 화면으로 복귀
- 로그인 상태 변화 없음

## 문제점 4: 구글/카카오 로그인 완료 후 엉뚱한 화면으로 이동됨

### 현상
앱에서 구글 또는 카카오 로그인을 하면, 완료 후 보관함 화면으로 이동됨.

### 원인
> 비유: 호텔 체크인을 마쳤는데 직원이 고객을 객실이 아닌 수영장으로 안내하는 상황

안드로이드 앱에서는 로그인 창을 새 팝업으로 열지 못합니다(안드로이드 보안 정책). 로그인 완료 페이지는 **독립적으로 열린 창**이며, 메인 앱은 백그라운드에 살아있습니다.

현재 코드에 **"보관함으로 이동"** 명령이 적혀있어, 이 명령이 메인 앱 전체를 보관함으로 강제 이동시킵니다.

강제로 메인화면이나 보관함으로 이동시키면 안 되는 이유:
- 메인화면 강제 이동 시: 카메라 초기화 코드가 다시 실행되어 권한 팝업 재발생
- 보관함 강제 이동 시: 사용자가 원하던 화면이 아닌 곳으로 이동

### 해결 방향
인증 완료 페이지는 독립 팝업이므로 **`window.close()` 단순 닫기**로 변경합니다.
창이 닫히면 메인 앱이 그대로 표시되고, 로그인된 상태로 사용자가 있던 화면(기능소개 화면)이 남아있습니다.

구글 로그인과 카카오 로그인 두 곳 모두 동일하게 적용.

### 수정 후 예상 시나리오
- 구글/카카오 로그인 완료: 인증 창이 닫힘 → 메인 앱이 그대로 표시됨
- 카메라 권한 팝업 재발생 없음
- 사용자가 있던 화면 상태 그대로 유지

## 문제점 5: 음성 읽기(TTS) 버튼이 기기마다 동작이 달라 불안정함

### 현상
AI 가이드 결과 화면에서 음성 재생 버튼을 눌렀을 때:
- 버튼이 "정지" 아이콘으로 바뀌는데 실제 음성은 안 나옴
- 또는 음성이 멈추지 않거나 다시 재생이 안 됨
- 기기마다 증상이 다름

### 원인
> 비유: 식당에서 주문 완료 팻말을 먼저 올려두고 요리를 시작하는 상황. 주방이 바빠서 요리를 못 만들어도 팻말은 그대로.

현재 코드는 텍스트 생성이 완료되면 **자동으로 음성 재생을 시작**하려 합니다. 그런데 안드로이드·iOS·PC 브라우저마다 "사용자가 직접 건드리지 않은 상황에서의 자동 음성 재생"을 차단하는 정도가 다릅니다.

- 어떤 기기: 자동 재생 허용
- 어떤 기기: 자동 재생 차단 (소리 없이 버튼만 "정지" 상태로 굳음)
- 어떤 기기: 재생 실패 신호를 보내지 않아 버튼이 영원히 "정지" 아이콘으로 남음

### 해결 방향 (핵심 변경)
**자동 재생을 완전히 제거하고, 사용자가 직접 버튼을 탭해야 재생되도록 변경합니다.**

새로운 흐름:
1. AI 텍스트 생성 완료 → 하단 첫 번째 버튼이 **삼각형(▶ 재생)** 아이콘으로 변경
2. 사용자가 삼각형 버튼을 탭 → 음성 재생 시작
3. 재생 중: 버튼이 **정지(⏸)** 아이콘으로 변경
4. 재생 완료: 버튼이 다시 **삼각형(▶)** 으로 복귀

이것이 기기별 자동 재생 차단을 우회하는 근본적인 해결책입니다. 사용자의 명시적 탭 동작이 있으므로 **모든 기기·모든 브라우저에서 100% 동작**합니다.

### 수정 후 예상 시나리오
- 텍스트 생성 완료: 삼각형 버튼이 즉시 표시됨 (자동 재생 없음)
- 사용자가 삼각형 버튼 탭: 즉시 음성 재생 시작
- 재생 완료: 삼각형 버튼으로 자동 복귀
- 모든 안드로이드·iOS·PC에서 동일하게 동작

## 문제점 6: 보관함·프로필·관리자 화면에서 음성 버튼이 안드로이드 Chrome에서 작동 안 함

### 현상
아래 화면들에서 음성 재생 버튼을 눌러도 음성이 멈추지 않거나, 멈춘 후 다시 재생이 안 됨:
- 보관함에서 저장된 가이드 열어볼 때
- 프로필 페이지에서 가이드 미리보기
- 관리자 대시보드에서 가이드 미리보기
- 다른 사람과 공유된 가이드북 페이지

### 원인
> 비유: 음악 플레이어의 일시정지 버튼이 안드로이드 폰에서는 전원 끄기처럼 동작해서, 다시 재생 버튼을 눌러도 곡이 처음부터 시작되지 않는 상황

안드로이드 Chrome 브라우저에는 **알려진 버그(Chromium 버그 #679437)**가 있습니다:
- 일반 브라우저(PC Chrome, iOS Safari): "일시정지" 명령이 정상 동작
- 안드로이드 Chrome: "일시정지" 명령이 음성을 완전히 꺼버리고 재생 불가 상태로 만듦

현재 4개 화면의 음성 중지 코드가 이 방식(일시정지 후 취소)을 사용하고 있습니다.
(참고: 메인 AI 결과 화면의 공유 페이지(`share-page.js`)는 이미 수정 완료됨)

### 해결 방향
4개 화면 모두에 동일한 수정을 적용합니다:
- **안드로이드 Chrome인지 자동 감지**
- 안드로이드 Chrome이면: 일시정지 건너뛰고 바로 중단 명령만 실행
- 그 외 기기(iOS, PC 등)는: 기존 방식 유지

### 수정 후 예상 시나리오
- 안드로이드 Chrome: 음성 재생 중 버튼 클릭 → 즉시 멈춤 → 다시 클릭 시 재생
- iOS, PC Chrome: 기존과 동일하게 작동
- 4개 화면 모두 동일한 동작 보장

**변경 내용:**

- 로그아웃 시 `localStorage.removeItem('cachedUser')`

**파일:** `public/index.js`

**변경 위치:** `queueForSpeech()` 함수

```javascript
변경 전:
function queueForSpeech(text, element) {
    utteranceQueue.push({ text, element });
    if (!isSpeaking) {
        speakNext();  // ← 자동 재생 시도

변경 후:
function queueForSpeech(text, element) {
```
